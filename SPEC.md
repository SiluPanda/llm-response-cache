# llm-response-cache -- Specification

## 1. Overview

`llm-response-cache` is a prompt-hash-keyed LLM response cache with model-version-aware invalidation and cost-savings tracking. It accepts complete LLM request parameters -- messages, model, temperature, and other generation settings -- hashes them into a deterministic cache key, and returns the cached response on a hit or delegates to the LLM on a miss. When a model version changes (OpenAI silently updates `gpt-4o` from one snapshot to another), the cache detects the version change and invalidates stale entries automatically. Every cache hit records the tokens and dollars saved, giving operators concrete visibility into how much the cache is reducing API spend.

The gap this package fills is specific and well-defined. General-purpose caches (`lru-cache`, `node-cache`, `keyv`) provide key-value storage with TTL and eviction, but they have no concept of LLM request structure. The caller must manually construct cache keys, decide which request parameters affect the output (temperature does, API key does not), handle model version changes, and track cost savings. This manual work is error-prone: forgetting to include `temperature` in the key means a `temperature=0` response can be served for a `temperature=1` request. Including the `stream` flag in the key means the same prompt with `stream: true` and `stream: false` produces two cache entries for the same output. Missing a model version change means serving stale responses from an older model snapshot.

`llm-response-cache` handles all of this automatically. It knows which parameters affect LLM output and which do not. It normalizes request parameters into a canonical form before hashing, so that `{ messages: [...], model: "gpt-4o", temperature: 0 }` and `{ temperature: 0, model: "gpt-4o", messages: [...] }` produce the same cache key. It tracks model versions from API response headers and invalidates entries when the model version changes. It estimates tokens saved per cache hit and computes cost savings using configurable model pricing.

This package is distinct from `llm-semantic-cache`, which uses embedding-based similarity matching to cache responses for semantically equivalent but textually different prompts. `llm-response-cache` uses exact hash matching -- two requests must have identical normalized parameters to produce the same cache key. Exact matching is simpler (no embeddings, no similarity computation, no threshold tuning), faster (hash computation in microseconds vs. embedding generation in milliseconds), cheaper (zero embedding API cost), and deterministic (no false positives from low thresholds, no false negatives from high thresholds). The tradeoff is coverage: exact matching misses paraphrased prompts. "What is the capital of France?" and "Tell me France's capital city" are different strings that produce different hashes, resulting in two cache entries for the same answer. For applications where prompts are generated by templates, come from deterministic code paths, or repeat verbatim (batch processing, dev/test workflows, idempotent API calls), exact matching achieves high hit rates at minimal cost. For applications with natural-language user input, semantic caching catches more duplicates. The two are complementary and can be layered: check the exact cache first (sub-millisecond, zero overhead), then check the semantic cache on an exact miss.

This package is also distinct from `embed-cache`, which caches embedding vectors for text inputs. `embed-cache` caches the output of embedding API calls (input: text, output: vector). `llm-response-cache` caches the output of LLM completion API calls (input: prompt + parameters, output: response text). They cache different artifacts at different layers of the AI stack.

## 2. Goals and Non-Goals

### Goals

- Provide a `createCache(options)` factory that returns a `ResponseCache` instance configured with a storage backend, eviction policy, model version tracking, and cost tracking settings.
- Provide a `cache.get(messages, model, params?)` method that normalizes the request parameters, computes the cache key hash, and returns the cached response on a hit or `null` on a miss.
- Provide a `cache.set(messages, model, params, response)` method that normalizes the request parameters, computes the cache key hash, and stores the response in the cache.
- Provide a `cache.wrap(client, options?)` method that returns a proxy LLM client where all chat/completion calls are transparently intercepted by the cache -- cache hits return immediately, cache misses call the underlying client and cache the response.
- Implement deterministic cache key construction from LLM request parameters. The key is `SHA-256(canonicalize(messages, model, temperature, top_p, max_tokens, tools, response_format, ...))`. Parameters that affect output are included; parameters that do not (API key, stream flag, request ID) are excluded.
- Implement stable normalization of request parameters before hashing: JSON serialization with sorted keys, whitespace trimming in message content, Unicode NFC normalization, removal of undefined/null fields. Two requests that are logically identical but differ in property order or whitespace produce the same hash.
- Implement model-version-aware cache invalidation. Cache entries record the model version (snapshot date or version identifier) from the API response. When the cache detects a model version change for a given model name, entries from the old version are flagged as stale and optionally auto-invalidated.
- Track model version information from LLM API response headers (`openai-model`, `x-model-version`, or equivalent) and from model name patterns (e.g., `gpt-4o-2024-08-06` contains an embedded date).
- Support configurable invalidation strategies: `auto` (stale entries are deleted immediately on version change detection), `soft` (stale entries are served with a warning flag, not deleted), `manual` (version tracking is recorded but no automatic invalidation occurs).
- Track cost savings on every cache hit: estimate tokens saved (input tokens from prompt, output tokens from cached response), compute cost saved using model-specific pricing. Maintain cumulative statistics.
- Provide `cache.stats()` returning `CacheStats` with hits, misses, hit rate, total tokens saved, total cost saved, entries by model, and model version information.
- Provide `cache.invalidate(model?)` to manually invalidate entries for a specific model or all models. Returns the count of invalidated entries.
- Provide `cache.clear()` and `cache.delete(key)` for cache management.
- Provide `cache.serialize()` and `ResponseCache.deserialize()` for exporting and importing cache state across environments.
- Support multiple storage backends via a `StorageBackend` interface: in-memory (LRU, default), filesystem (JSON files), SQLite (persistent, queryable), Redis (distributed), and custom adapters.
- Support eviction policies: TTL (time-based expiry), LRU (least recently used), max size (entry count limit), and model-version-based (entries from superseded model versions are evicted first).
- Support streaming response caching: on a cache miss with `stream: true`, the stream is forwarded to the caller in real time while the full response is buffered and cached on completion. On a cache hit with `stream: true`, the cached response is replayed as a synthetic stream.
- Zero runtime dependencies for the core in-memory mode. Optional peer dependencies for storage backends (`better-sqlite3`, `ioredis`).
- Target Node.js 18+. Use `node:crypto` for SHA-256 hashing, built-in `node:util` for CLI argument parsing.

### Non-Goals

- **Not a semantic cache.** This package uses exact hash matching, not embedding-based similarity. "What is the capital of France?" and "Tell me France's capital city" are different strings with different hashes -- they produce separate cache entries. For semantic caching that catches paraphrases, use `llm-semantic-cache`. The two packages are complementary and can be layered.
- **Not an embedding cache.** This package caches complete LLM responses, not embedding vectors. For caching embedding API calls, use `embed-cache`.
- **Not an LLM provider or proxy.** This package intercepts LLM calls to check the cache but does not implement any LLM API. It does not route requests, load balance, retry on failure, or manage rate limits. Use `llm-failover`, `llm-retry`, or a dedicated LLM gateway for those concerns.
- **Not a prompt normalization tool.** While the cache normalizes request parameters for key computation (whitespace trimming, JSON key sorting), it does not perform semantic prompt normalization (synonym replacement, template extraction, rewriting). For prompt-level normalization that increases cache hit rates without embeddings, use `prompt-dedup` as a preprocessing step.
- **Not a distributed cache coordinator.** The Redis backend enables shared caching across multiple processes, but `llm-response-cache` does not implement distributed locking, write conflict resolution, or cache invalidation protocols. Two processes may simultaneously miss the same prompt and both call the LLM, with both writing the same response. This is safe (idempotent writes) but wastes an LLM call. For high-concurrency deduplication of in-flight requests, combine with `llm-dedup`.
- **Not a model version registry.** The cache tracks model versions observed in API responses for invalidation purposes. It does not maintain a comprehensive registry of all model versions, release dates, or changelog information. It observes what the API reports and acts on changes.
- **Not a cost management platform.** Cost tracking is a reporting feature, not a control feature. The cache reports how much it has saved but does not enforce budgets, throttle requests at cost thresholds, or alert on spending. For budget enforcement, integrate with a separate cost management layer.

## 3. Target Users and Use Cases

### Batch Processing Engineers

Teams running batch processing pipelines that send the same prompts repeatedly. An evaluation pipeline that tests 500 prompts against an LLM, runs 10 times per day during development, generates 5,000 LLM calls per day. If the prompts and parameters are identical across runs, every call after the first is a cache hit. At $0.005 per call (GPT-4o, 500-token response), the cache saves $22.50/day -- $8,212/year. Batch processing is the ideal use case for exact-match caching because prompts are generated programmatically and repeat verbatim.

### Development and Testing Workflows

Developers iterating on LLM-powered applications during development. Every code change triggers a test run that calls the LLM with the same test prompts. Without caching, each test run accumulates API cost and is gated by API latency (500ms-5s per call). With a persistent response cache (SQLite backend), test prompts that were called previously return instantly with zero API cost, even across process restarts. A developer running tests 50 times per day with 20 test prompts saves 1,000 API calls per day. With a local SQLite cache, test suite latency drops from minutes to milliseconds for cached prompts.

### Template-Based Prompt Systems

Applications that construct prompts from templates with a finite set of variable values. A product recommendation system that fills a template with product categories: "Recommend 5 products in the {category} category for a {age_group} audience." If there are 50 categories and 5 age groups, there are 250 unique prompts. After all 250 have been generated once, every subsequent request is a cache hit. Template-based prompts are deterministic and repeat exactly, making exact-match caching achieve near-100% hit rates after warmup.

### Cost Optimization Engineers

Engineers tasked with reducing LLM API spend. The cache's `stats()` method returns `{ hits, misses, hitRate, tokensSaved, costSaved }` with a concrete dollar figure. Combined with `model-price-registry` for accurate per-model pricing, this enables data-driven cost optimization: "The response cache saved $4,200 last month by eliminating 840,000 redundant API calls." The cost tracking is not a guess -- it is computed from actual cache hits, cached response token counts, and model pricing.

### CI/CD Pipeline Operators

Teams running LLM-based validation, content generation, or testing in CI/CD pipelines. Each CI run sends prompts to the LLM for tasks like code review, documentation generation, or test case generation. Prompts for unchanged code are identical across runs. A shared Redis-backed response cache across CI workers ensures that the first worker to process a prompt populates the cache, and subsequent workers hit the cache. CI pipeline latency drops and API costs are amortized across the team.

### Idempotent API Call Systems

Applications where LLM calls are idempotent -- the same input always should produce the same output. Structured data extraction (extracting entities from text), classification tasks (categorizing support tickets), and deterministic analysis (code review for a specific diff) all produce the same output for the same input when temperature is 0. These are ideal for exact-match caching because the output is deterministic and any repeat call is pure waste.

### Multi-Environment Development Teams

Teams working across local, staging, and production environments. A response cache serialized from production can be imported into staging or local environments, providing realistic cached responses without API access. Developers working offline can iterate using cached responses from previous sessions. The `serialize()` / `deserialize()` methods enable cache portability across environments.

## 4. Core Concepts

### Cache Key

A cache key is a deterministic identifier derived from the LLM request parameters that affect the output. Two requests with the same cache key produce the same LLM response (assuming deterministic model behavior, i.e., temperature=0 or the same seed). The key is a SHA-256 hash of the canonicalized request parameters.

The cache key is the fundamental unit of cache identity. If two requests produce the same key, they are cache-equivalent -- the cached response for one is valid for the other. If they produce different keys, they are distinct requests that may produce different responses.

The key is not the raw request object. It is the hash of a normalized, canonicalized representation of the output-affecting parameters. This normalization ensures that logically identical requests produce the same key regardless of property order, whitespace variation, or inclusion of output-irrelevant fields.

### Prompt Hash

The prompt hash is the SHA-256 digest of the canonicalized request parameters. SHA-256 is chosen for three reasons:

1. **Collision resistance.** The probability of two different canonicalized inputs producing the same 256-bit hash is astronomically low (birthday bound: ~2^128 operations). For a cache with millions of entries, the probability of a collision is effectively zero.
2. **Determinism.** The same input always produces the same hash, across platforms, Node.js versions, and process restarts.
3. **Speed.** SHA-256 hashing of a typical LLM request (1-10 KB of serialized JSON) takes under 10 microseconds on modern hardware using `node:crypto`.

The hash is represented as a lowercase hex string (64 characters) and serves as both the cache key and the storage key in all backends.

### Model Versioning

LLM providers update models without changing the model name. OpenAI has updated `gpt-4o` from the `gpt-4o-2024-05-13` snapshot to `gpt-4o-2024-08-06` to `gpt-4o-2024-11-20` while all three are accessible via the `gpt-4o` alias. Anthropic updates Claude models with new training data and behavior changes. When the underlying model changes, cached responses from the previous version may be incorrect, outdated, or stylistically inconsistent with what the current model would produce.

Model-version-aware caching addresses this by recording the model version at the time of caching and detecting when the version changes. The version is extracted from:

1. **API response headers.** OpenAI includes the specific model snapshot in the response object's `model` field (e.g., the request specifies `gpt-4o` but the response reports `gpt-4o-2024-08-06`). Anthropic includes version information in response metadata.
2. **Model name patterns.** Model names with embedded dates (e.g., `gpt-4o-2024-08-06`, `claude-3-5-sonnet-20241022`) contain explicit version information.
3. **Manual configuration.** The caller can specify known model versions via the cache configuration.

When the cache observes a response from a model version that differs from the version recorded in existing cache entries for that model, it triggers version-change handling according to the configured invalidation strategy.

### Cache Hit and Cache Miss

A **cache hit** occurs when the computed cache key matches an existing, non-expired, non-invalidated cache entry. The cached response is returned immediately without calling the LLM API. Cache hits are the source of all time and cost savings.

A **cache miss** occurs when the computed cache key does not match any entry, or the matching entry is expired (TTL exceeded), or the matching entry is invalidated (model version change). On a miss, the request is forwarded to the LLM API, the response is cached, and the response is returned to the caller.

### Cost Savings

Every cache hit avoids one LLM API call. The cost of that avoided call is computed from:

- **Input tokens saved:** The token count of the prompt messages that would have been sent.
- **Output tokens saved:** The token count of the response that would have been generated.
- **Model pricing:** The price per million tokens for the model (input price for prompt tokens, output price for response tokens).

The formula for cost saved per hit:

```
costSaved = (inputTokens / 1_000_000 * inputPricePerMTok) + (outputTokens / 1_000_000 * outputPricePerMTok)
```

Token counts are estimated from text length using a configurable estimator (default: 4 characters per token for English text). For precise tracking, the cached response can store the actual token counts from the original API response's `usage` field.

## 5. Cache Key Construction

### Overview

The cache key is the SHA-256 hash of a canonical JSON representation of the request parameters that affect the LLM output. Two requests that differ only in output-irrelevant parameters (API key, request ID, stream flag) must produce the same key. Two requests that differ in any output-relevant parameter (messages, model, temperature) must produce different keys.

### Canonicalization Steps

The canonicalization process transforms a raw LLM request into a deterministic string suitable for hashing:

**Step 1: Extract output-affecting parameters.** From the request object, extract only the parameters that influence the LLM response content. Discard all others.

**Step 2: Normalize message content.** For each message in the messages array:
- Apply Unicode NFC normalization to the `content` field.
- Trim leading and trailing whitespace from the `content` field.
- Preserve the `role` field exactly as provided.
- Preserve the order of messages (order affects output).
- For multi-part content (arrays of text and image parts), normalize each text part individually. Image parts are included by reference (URL) or by content hash.

**Step 3: Normalize the model identifier.** Convert the model name to lowercase. `GPT-4o` becomes `gpt-4o`.

**Step 4: Normalize numeric parameters.** Convert `temperature`, `top_p`, `max_tokens`, `frequency_penalty`, `presence_penalty`, and `seed` to their canonical numeric representations. `temperature: 0` and `temperature: 0.0` produce the same canonical value.

**Step 5: Serialize with sorted keys.** Serialize the normalized parameter object as JSON with keys sorted alphabetically at every level of nesting. This eliminates key-order dependency: `{ model: "gpt-4o", temperature: 0 }` and `{ temperature: 0, model: "gpt-4o" }` produce the same string.

**Step 6: Hash.** Compute `SHA-256` of the canonical JSON string. Represent as a lowercase hex string.

### Parameters Included in the Key

| Parameter | Why It Affects Output |
|---|---|
| `messages` | The prompt content. Different messages produce different responses. |
| `model` | Different models produce different responses. |
| `temperature` | Controls randomness. temperature=0 is deterministic; temperature=1 is creative. |
| `top_p` | Nucleus sampling threshold. Affects which tokens are considered. |
| `max_tokens` | Limits response length. A 100-token cap may truncate a response that a 1000-token cap would complete. |
| `frequency_penalty` | Penalizes repeated tokens. Affects response style and content. |
| `presence_penalty` | Penalizes tokens that have appeared. Affects response breadth. |
| `seed` | When provided, makes sampling deterministic for a given seed value. |
| `tools` / `functions` | Available tools change the model's behavior and response format. |
| `tool_choice` / `function_call` | Forces or restricts tool use, changing response content. |
| `response_format` | Constrains output structure (JSON mode, structured output schema). |
| `stop` | Stop sequences affect where the response ends. |
| `logit_bias` | Biases specific tokens, changing response content. |

### Parameters Excluded from the Key

| Parameter | Why It Does Not Affect Output |
|---|---|
| `stream` | Controls delivery format (chunked vs. complete), not content. |
| `stream_options` | Stream configuration (include_usage, etc.) does not affect content. |
| `n` | Number of completions. The cache stores one response per key. |
| `user` | User identifier for abuse monitoring. Does not affect model output. |
| `api_key` / auth headers | Authentication. Does not affect model output. |
| `timeout` | Client-side timeout. Does not affect model output. |
| `request_id` / `idempotency_key` | Request tracking. Does not affect model output. |
| `organization` | OpenAI org identifier. Does not affect model output. |

### Why Each Included Parameter Matters

**`temperature`** is the most commonly mishandled parameter. A response cached at `temperature=0` (deterministic, focused) must not be returned for a request at `temperature=1` (random, creative). If temperature were excluded from the key, the first request for a prompt would populate the cache, and all subsequent requests with different temperatures would incorrectly receive the first response.

**`max_tokens`** matters because it determines response completeness. A prompt cached with `max_tokens=50` may produce a truncated response. If the same prompt is later requested with `max_tokens=500`, the truncated cached response would be incorrect. Including `max_tokens` in the key ensures that requests with different length constraints are treated as different cache entries.

**`tools`** / **`response_format`** matter because they fundamentally change the model's output. A prompt without tools produces a text response. The same prompt with tools may produce a tool call. A prompt without `response_format` produces free-form text. With `response_format: { type: "json_object" }`, it produces JSON. These are not cosmetic differences -- they are structural changes to the response.

### Default Parameter Values

When a parameter is not specified in the request, it takes the model's default value. The cache must treat absent parameters consistently:

- If `temperature` is absent, it is treated as absent (not as `0` or `1`). The canonical form omits the key entirely. This means a request with `temperature: undefined` matches another request with `temperature` omitted.
- Explicitly setting a parameter to `undefined` or `null` is equivalent to omitting it.
- Explicitly setting a parameter to its default value (e.g., `temperature: 1` for OpenAI) is NOT equivalent to omitting it, because the cache cannot assume it knows every model's default. The canonical form includes explicitly set values even if they match the default.

This design is conservative: it may produce separate cache entries for `{ temperature: 1 }` and `{}` (where the model default is 1), but it never produces incorrect cache hits.

### Hash Computation

```typescript
import { createHash } from 'node:crypto';

function computeCacheKey(params: CanonicalizableParams): string {
  const canonical = canonicalize(params);
  return createHash('sha256').update(canonical).digest('hex');
}
```

The hash is computed once per cache lookup and once per cache store. The computation takes under 10 microseconds for a typical request (1-10 KB serialized JSON). This is negligible compared to LLM API call latency (500ms-5s).

## 6. Model-Version-Aware Invalidation

### The Problem

LLM providers update models continuously. OpenAI maintains model aliases (`gpt-4o`) that resolve to specific snapshots (`gpt-4o-2024-08-06`). When OpenAI updates the alias to point to a newer snapshot, the behavior of the model changes. Responses cached from the old snapshot may be factually outdated, stylistically different, or structurally incompatible with the new snapshot.

Without version-aware invalidation, a cache accumulates stale entries indefinitely. A response cached when `gpt-4o` resolved to `gpt-4o-2024-05-13` is served months later when `gpt-4o` resolves to `gpt-4o-2024-11-20`. The user expects the new model's behavior but receives the old model's cached response.

### Version Detection

The cache detects model versions from two sources:

**Source 1: API response metadata.** When the LLM responds to a cache miss, the response includes the resolved model version. For OpenAI, the `ChatCompletion` object contains a `model` field with the specific snapshot (e.g., `gpt-4o-2024-08-06` even though the request specified `gpt-4o`). The cache extracts this resolved model identifier and stores it alongside the cached response.

```typescript
// Request specifies alias
const request = { model: 'gpt-4o', messages: [...] };

// Response resolves to snapshot
const response = await openai.chat.completions.create(request);
// response.model === 'gpt-4o-2024-08-06'

// Cache stores both the requested model ('gpt-4o') and the resolved version ('gpt-4o-2024-08-06')
```

**Source 2: Model name patterns.** Model names with embedded dates (e.g., `gpt-4o-2024-08-06`, `claude-3-5-sonnet-20241022`) contain explicit version information. The cache extracts the date component as a version identifier. When the caller specifies a dated model name directly, the version is known at request time without needing to inspect the response.

### Version Tracking

The cache maintains a model version registry -- a mapping from model aliases to their most recently observed resolved versions:

```
{
  "gpt-4o": "gpt-4o-2024-11-20",
  "gpt-4o-mini": "gpt-4o-mini-2024-07-18",
  "gpt-4-turbo": "gpt-4-turbo-2024-04-09"
}
```

This registry is updated on every cache miss (when the LLM API responds with a resolved model version). When the resolved version for a model alias changes, the cache triggers a version change event.

### Invalidation Strategies

Three invalidation strategies control what happens when a version change is detected:

**`auto` (default):** All cache entries for the affected model alias that were cached under the old version are immediately deleted. This is the safest strategy -- it guarantees that no stale responses are served. The downside is that it discards the cache, forcing re-population from scratch. For applications where response correctness is paramount, use `auto`.

**`soft`:** Entries from the old version are not deleted but are flagged as stale. When a stale entry is served as a cache hit, the response includes a `stale: true` flag and a `staleReason` string. The caller can inspect these flags and decide whether to accept the stale response or re-fetch from the LLM. This is useful for applications that prefer a possibly-outdated response over a slow response, with the option to refresh asynchronously.

```typescript
const result = cache.get(messages, model, params);
if (result && result.stale) {
  console.warn(`Stale cache hit: ${result.staleReason}`);
  // Optionally re-fetch in background
}
```

**`manual`:** Version tracking is recorded but no automatic invalidation occurs. The cache serves entries regardless of version changes. The caller must explicitly call `cache.invalidate(model)` to remove stale entries. This gives the caller full control over invalidation timing, useful for applications with custom staleness policies.

### Version Change Detection Flow

```
1. Application calls cache.wrap(client) or cache.set() after a cache miss.
2. LLM response includes resolved model version (e.g., 'gpt-4o-2024-11-20').
3. Cache checks version registry: last known version for 'gpt-4o' was 'gpt-4o-2024-08-06'.
4. Version mismatch detected → version change event.
5. Registry updated: 'gpt-4o' → 'gpt-4o-2024-11-20'.
6. Based on invalidation strategy:
   - auto: delete all 'gpt-4o' entries with version 'gpt-4o-2024-08-06'.
   - soft: flag all 'gpt-4o' entries with version 'gpt-4o-2024-08-06' as stale.
   - manual: no action, caller must call cache.invalidate('gpt-4o').
7. New response is cached with version 'gpt-4o-2024-11-20'.
```

### Dated Model Names

When the caller specifies a dated model name directly (e.g., `gpt-4o-2024-08-06`), version tracking is straightforward. The model name IS the version. If the caller later switches to `gpt-4o-2024-11-20`, the old entries under `gpt-4o-2024-08-06` are not invalidated because they are in a separate key space (the model name is part of the cache key). This is the correct behavior: the caller explicitly requested a specific version, and the cached responses are valid for that version.

Version invalidation only triggers when the resolved version of an alias changes, not when different explicit versions coexist. `gpt-4o-2024-08-06` and `gpt-4o-2024-11-20` are separate cache namespaces. `gpt-4o` is an alias namespace whose underlying version may change.

## 7. Cache-Through Wrapper

### `cache.wrap(client, options?)`

The cache-through wrapper is the primary integration mechanism. It accepts an LLM client object and returns a proxy object with the same interface where all chat/completion calls are intercepted by the cache. Application code uses the wrapped client exactly as it would use the original client -- the cache is transparent.

```typescript
import { createCache } from 'llm-response-cache';
import OpenAI from 'openai';

const openai = new OpenAI();
const cache = createCache({
  storage: 'memory',
  maxEntries: 10_000,
});

const cachedClient = cache.wrap(openai);

// This call goes through the cache transparently
const response = await cachedClient.chat.completions.create({
  model: 'gpt-4o',
  temperature: 0,
  messages: [{ role: 'user', content: 'What is the capital of France?' }],
});
// First call: cache miss → calls OpenAI → caches response → returns
// Second identical call: cache hit → returns cached response (<1ms)
```

### How Wrapping Works

The wrapper uses a JavaScript `Proxy` to intercept method calls on the client object. When `chat.completions.create` (OpenAI-style) or `messages.create` (Anthropic-style) is called, the wrapper:

1. Extracts the request parameters (messages, model, temperature, etc.) from the call arguments.
2. Computes the cache key by canonicalizing the output-affecting parameters and hashing with SHA-256.
3. Checks the cache for a hit using the computed key.
4. On hit: constructs a response object matching the original client's response format, records the cache hit in statistics, and returns the cached response.
5. On miss: delegates to the original client method, extracts the model version from the response, caches the response along with version metadata, records the cache miss in statistics, and returns the response.

### Client Detection

The wrapper automatically detects the client type:

- **OpenAI SDK:** Detected by the presence of `client.chat.completions.create`. Intercepts `chat.completions.create`.
- **Anthropic SDK:** Detected by the presence of `client.messages.create`. Intercepts `messages.create`.
- **Custom clients:** The caller provides `extractParams`, `buildResponse`, and method path via the `WrapOptions` configuration.

### Response Format Reconstruction

When returning a cached response, the wrapper constructs a response object that matches the format the original client would return:

**OpenAI format:**
```typescript
{
  id: 'cache-<hash>',
  object: 'chat.completion',
  created: Math.floor(Date.now() / 1000),
  model: cachedEntry.resolvedModel,
  choices: [{
    index: 0,
    message: { role: 'assistant', content: cachedEntry.responseText },
    finish_reason: cachedEntry.finishReason || 'stop',
  }],
  usage: { prompt_tokens: 0, completion_tokens: 0, total_tokens: 0 },
  _cached: true,
  _cacheKey: cachedEntry.key,
  _cachedAt: cachedEntry.createdAt,
}
```

**Anthropic format:**
```typescript
{
  id: 'cache-<hash>',
  type: 'message',
  role: 'assistant',
  content: [{ type: 'text', text: cachedEntry.responseText }],
  model: cachedEntry.resolvedModel,
  usage: { input_tokens: 0, output_tokens: 0 },
  stop_reason: cachedEntry.finishReason || 'end_turn',
  _cached: true,
  _cacheKey: cachedEntry.key,
  _cachedAt: cachedEntry.createdAt,
}
```

The `_cached: true` property is added to all cached responses, allowing downstream code to detect cache hits programmatically. The `usage` field is set to zero because no tokens were consumed.

### Streaming Support

The cache-through wrapper handles streaming responses (`stream: true` in the request parameters).

**On a cache miss with streaming:**

1. The wrapper delegates to the original client with `stream: true`.
2. It returns an async iterable to the caller that yields chunks as they arrive from the LLM.
3. Internally, it buffers the full response content as chunks arrive.
4. When the stream completes (final chunk received), the wrapper assembles the full response from the buffer and caches it.
5. The caller receives the stream in real time -- there is no added latency.

**On a cache hit with streaming:**

1. The wrapper finds the cached response.
2. It returns an async iterable that yields the cached response as synthetic chunks.
3. Three replay modes are available:
   - **`immediate` (default):** The full response is emitted as a single chunk, followed by a completion signal. Total replay time: sub-1ms.
   - **`simulated`:** The response is split into token-sized chunks and emitted with small delays to simulate natural LLM generation speed.
   - **`throttled`:** The response is emitted at a configurable tokens-per-second rate.

**Response integrity:** The cache stores only fully completed responses. If a stream is interrupted before completion (client disconnect, timeout, network error), nothing is cached. The next request for the same prompt results in a cache miss.

## 8. Cost-Savings Tracking

### What Is Tracked

The cache maintains running counters for cost-related metrics:

- **`hits`**: Total number of cache lookups that returned a cached response.
- **`misses`**: Total number of cache lookups that resulted in an LLM API call.
- **`hitRate`**: `hits / (hits + misses)`. A value between 0 and 1.
- **`tokensSaved`**: Estimated total tokens saved by cache hits. For each hit, the token count is computed from the cached response's stored token usage (if available from the original API response) or estimated from text length.
- **`costSaved`**: Estimated total USD saved by cache hits. Computed per-hit as `(inputTokens * inputPrice + outputTokens * outputPrice) / 1_000_000`.
- **`entriesByModel`**: Number of cached entries per model.
- **`modelVersions`**: Currently tracked model version registry.

### Per-Hit Cost Computation

When a cache hit occurs, the cost saved is computed as follows:

```
inputTokens  = cachedEntry.usage?.prompt_tokens     ?? estimateTokens(requestMessages)
outputTokens = cachedEntry.usage?.completion_tokens  ?? estimateTokens(cachedEntry.responseText)
inputPrice   = modelPrices[model]?.input  ?? defaultPrice.input
outputPrice  = modelPrices[model]?.output ?? defaultPrice.output

costSavedThisHit = (inputTokens / 1_000_000 * inputPrice) + (outputTokens / 1_000_000 * outputPrice)
```

The cache stores the original API response's `usage` field (which contains exact token counts) in the cache entry. When available, exact token counts are used for cost computation. When not available (e.g., entries imported from another system or entries where usage was not captured), the estimator falls back to character-based approximation.

### Token Estimation

The default token estimator uses the approximation that 1 token is approximately 4 characters for English text. This is a commonly used heuristic based on the observation that GPT tokenizers produce approximately 1 token per 4 characters for typical English prose. The estimator is configurable:

```typescript
const cache = createCache({
  tokenEstimator: (text) => Math.ceil(text.length / 4),
  // Or use a precise tokenizer:
  // tokenEstimator: (text) => encode(text).length, // using tiktoken
});
```

### Integration with model-price-registry

`llm-response-cache` integrates with `model-price-registry` from this monorepo for accurate, up-to-date model pricing:

```typescript

    'gpt-4o': getPrice('gpt-4o'),              // { input: 2.50, output: 10.00 }
```

If `model-price-registry` is not installed, the cache uses built-in default prices for common models and falls back to `{ input: 1.00, output: 2.00 }` per million tokens for unknown models.

## 9. Storage Backends

### In-Memory (Default)

The default backend stores cache entries in a JavaScript `Map` keyed by the SHA-256 hash. An LRU doubly-linked list maintains access order for eviction.

**Characteristics:**
- Data is lost when the process exits.
- Memory usage: each cache entry stores the response text, request metadata, and version information. A typical entry (500-token response, metadata) is approximately 3-5 KB. A cache with 10,000 entries uses approximately 30-50 MB.
- Read/write latency: sub-microsecond for individual operations.
- Suitable for: development, testing, short-lived processes, applications where cache persistence is not needed.

**Configuration:**
```typescript
createCache({ storage: 'memory' }) // default
createCache({ storage: { type: 'memory' } })
```

### Filesystem (JSON)

The filesystem backend stores cache entries as a JSON file on disk. On initialization, the full file is loaded into memory. Writes are debounced (default: 1 second) to avoid excessive disk I/O.

**Characteristics:**
- Persists across process restarts.
- JSON serialization is verbose: response text and metadata as JSON can be large for caches with many entries.
- Startup latency: depends on file size (approximately 1-3 seconds for a 50 MB file).
- Not suitable for multi-process environments (no write locking).
- Suitable for: development, CI, simple single-process deployments.

**Configuration:**
```typescript
createCache({
  storage: {
    type: 'filesystem',
    path: './cache/response-cache.json',
    debounceMs: 1000, // default: 1000
  },
})
```

### SQLite

The SQLite backend stores cache entries in a single-file SQLite database using `better-sqlite3` (peer dependency). The schema:

```sql
CREATE TABLE IF NOT EXISTS cache_entries (
  key TEXT PRIMARY KEY,
  request_hash TEXT NOT NULL,
  messages TEXT NOT NULL,
  model TEXT NOT NULL,
  resolved_model TEXT,
  model_version TEXT,
  params TEXT NOT NULL,
  response TEXT NOT NULL,
  response_text TEXT NOT NULL,
  usage TEXT,
  finish_reason TEXT,
  created_at INTEGER NOT NULL,
  accessed_at INTEGER NOT NULL,
  hit_count INTEGER NOT NULL DEFAULT 0,
  ttl INTEGER,
  stale INTEGER NOT NULL DEFAULT 0,
  stale_reason TEXT,
  metadata TEXT
);

CREATE INDEX IF NOT EXISTS idx_model ON cache_entries(model);
CREATE INDEX IF NOT EXISTS idx_model_version ON cache_entries(model_version);
CREATE INDEX IF NOT EXISTS idx_accessed_at ON cache_entries(accessed_at);
CREATE INDEX IF NOT EXISTS idx_created_at ON cache_entries(created_at);
CREATE INDEX IF NOT EXISTS idx_stale ON cache_entries(stale);
```

**Characteristics:**
- Persists across process restarts. Data survives crashes (ACID with WAL mode).
- Supports concurrent reads (WAL mode enabled by default).
- Write latency: approximately 0.5-2ms per entry.
- Read latency: approximately 0.1-0.5ms per entry.
- Supports efficient queries for model-version-based invalidation (`UPDATE ... WHERE model = ? AND model_version = ?`).
- Suitable for: single-machine production deployments, persistent development caches, moderate-scale caches (up to ~100,000 entries).
- Peer dependency: `better-sqlite3`.

**Configuration:**
```typescript
createCache({
  storage: {
    type: 'sqlite',
    path: './cache/response-cache.db',
    walMode: true, // default: true
  },
})
```

### Redis

The Redis backend stores cache entries in Redis using `ioredis` (peer dependency). Each entry is stored as a Redis hash with fields for request hash, messages, model, response, timestamps, and metadata.

**Characteristics:**
- Shared across multiple processes and machines.
- Redis TTL support (entries expire automatically via Redis `PEXPIRE`).
- Write latency: 1-5ms (network round trip).
- Read latency: 1-5ms (network round trip).
- Suitable for: distributed deployments, multi-process applications, shared caches across CI workers.
- Peer dependency: `ioredis`.

**Configuration:**
```typescript
createCache({
  storage: {
    type: 'redis',
    url: 'redis://localhost:6379',
    keyPrefix: 'resp-cache:', // default
  },
})
```

**Multi-process behavior:** Multiple processes sharing the same Redis instance share the response cache. When one process caches a response, all other processes can immediately read it. Cache keys are deterministic (same request → same hash), so concurrent processes naturally converge on the same cache entries.

### Custom Backend

Any object implementing the `StorageBackend` interface can be used:

```typescript
interface StorageBackend {
  /** Get a cache entry by key. Returns undefined if not found or expired. */
  get(key: string): Promise<CacheEntry | undefined>;

  /** Store a cache entry. */
  set(entry: CacheEntry): Promise<void>;

  /** Delete an entry by key. Returns true if the entry existed. */
  delete(key: string): Promise<boolean>;

  /** Clear all entries, optionally filtered by model. */
  clear(options?: { model?: string }): Promise<void>;

  /** Return entries matching a filter. Used for model-version invalidation. */
  query(filter: { model?: string; modelVersion?: string; stale?: boolean }): Promise<CacheEntry[]>;

  /** Update entries matching a filter. Used for soft invalidation. */
  update(filter: { model?: string; modelVersion?: string }, update: Partial<CacheEntry>): Promise<number>;

  /** Return the total number of entries. */
  size(): Promise<number>;

  /** Return the number of entries per model. */
  sizeByModel(): Promise<Map<string, number>>;

  /** Optional: flush any pending writes to durable storage. */
  flush?(): Promise<void>;

  /** Optional: close any open connections or file handles. */
  close?(): Promise<void>;
}
```

The `query` and `update` methods are required for model-version-aware invalidation. Backends that cannot support these operations (simple key-value stores) can implement them as full-scan + filter operations over the entry set.

## 10. Eviction Policies

### TTL (Time-to-Live)

Entries expire after a configurable duration. Expired entries are not returned on cache hits and are lazily removed during access or eagerly removed by a background sweep.

```typescript
createCache({
  ttl: 3600_000, // 1 hour in milliseconds
})
```

TTL is checked on every cache hit. If an entry's `createdAt + ttl < now`, the entry is treated as expired: the cache returns a miss and schedules the entry for deletion.

TTL can be set globally (all entries share the same TTL) or per entry (via `cache.set(messages, model, params, response, { ttl: 1800_000 })`). Per-entry TTL overrides the global TTL.

**When to use TTL:** When LLM responses become stale over time. A cache of responses about current events should expire after hours. A cache of responses about factual knowledge (math, geography) can have a long TTL or no TTL at all. A cache used during development might have no TTL, relying on manual invalidation.

### LRU (Least Recently Used)

When the cache reaches its maximum size (`maxEntries`), the least recently used entry is evicted to make room for new entries. "Recently used" is determined by the `accessedAt` timestamp, which is updated on every cache hit.

```typescript
createCache({
  maxEntries: 10_000,
  evictionPolicy: 'lru', // default when maxEntries is set
})
```

LRU eviction is implemented differently by backend:

- **In-memory:** A doubly-linked list maintains access order. Hit promotion and eviction are O(1).
- **SQLite:** Eviction queries `DELETE FROM cache_entries ORDER BY accessed_at ASC LIMIT ?` to remove the oldest-accessed entries.
- **Redis:** Entries are tracked in a Redis sorted set keyed by access timestamp. Eviction removes entries with the lowest scores.

### Max Entries

A hard limit on the total number of cache entries. When the limit is reached, the eviction policy determines which entry to remove. If no eviction policy is explicitly configured, `lru` is used by default when `maxEntries` is set.

```typescript
createCache({
  maxEntries: 50_000,
})
```

### Model-Version-Based Eviction

Entries from superseded model versions are evicted preferentially. When space is needed, entries flagged as stale (from old model versions) are evicted before non-stale entries. This policy is automatic when `invalidationStrategy: 'soft'` is configured: stale entries serve as a buffer that can be evicted without losing current-version entries.

### Combined Policies

TTL and LRU/max entries can be combined. An entry is evicted if either condition is met: TTL expired, LRU eviction triggered by size limit, or model-version staleness.

```typescript
createCache({
  ttl: 86400_000,       // expire after 24 hours
  maxEntries: 50_000,   // also limit to 50,000 entries
  evictionPolicy: 'lru',
  invalidationStrategy: 'soft', // stale entries evicted first
})
```

## 13. Integration with Monorepo Packages

### llm-semantic-cache

`llm-semantic-cache` provides semantic (similarity-based) LLM response caching. `llm-response-cache` provides exact (hash-based) caching. The two are complementary and should be layered: check the exact cache first (sub-millisecond, zero overhead), then check the semantic cache on an exact miss (requires embedding, catches paraphrases).

```typescript
import { createCache as createExactCache } from 'llm-response-cache';
import { createCache as createSemanticCache } from 'llm-semantic-cache';

const exactCache = createExactCache({
  maxEntries: 50_000,
  storage: { type: 'sqlite', path: './exact-cache.db' },
});

const semanticCache = createSemanticCache({
  embedder: myEmbedder,
  threshold: 0.92,
  maxEntries: 10_000,
  storage: { type: 'sqlite', path: './semantic-cache.db' },
});

async function cachedLLM(messages: Message[], model: string, params: Params): Promise<string> {
  // Layer 1: Exact match (sub-1ms, no embedding cost)
  const exact = exactCache.get(messages, model, params);
  if (exact) return exact.response;

  // Layer 2: Semantic match (2-10ms with local embedder)
  const prompt = messagesToString(messages);
  const semantic = await semanticCache.get(prompt, { model });
  if (semantic) return semantic.response;

  // Layer 3: LLM call (500-5000ms, full cost)
  const response = await callLLM(messages, model, params);
  exactCache.set(messages, model, params, { text: response, resolvedModel: model });
  await semanticCache.set(prompt, response, { model });
  return response;
}
```

### embed-cache

`embed-cache` caches embedding vectors. `llm-response-cache` caches LLM responses. They operate at different layers and do not directly interact. However, in a layered caching architecture (exact cache + semantic cache), `embed-cache` can wrap the embedding function used by `llm-semantic-cache` to avoid redundant embedding API calls. The three form a complete caching stack:

1. `llm-response-cache` -- caches exact-match LLM responses (fastest, cheapest check)
2. `llm-semantic-cache` -- caches semantically similar LLM responses (catches paraphrases)
3. `embed-cache` -- caches embedding vectors used by the semantic cache (reduces embedding API calls)

### model-price-registry

`model-price-registry` provides up-to-date pricing for LLM models. `llm-response-cache` uses it for cost tracking in the `stats()` method:

```typescript
import { getPrice } from 'model-price-registry';
import { createCache } from 'llm-response-cache';

const cache = createCache({
  modelPrices: {
    'gpt-4o': getPrice('gpt-4o'),
    'gpt-4o-mini': getPrice('gpt-4o-mini'),
  },
});
```

### llm-vcr

`llm-vcr` records and replays LLM API interactions for testing. `llm-response-cache` can serve a similar purpose for deterministic prompts: a pre-populated cache acts as a replay layer. The difference is intent -- `llm-vcr` is designed for test fixture management with assertion support, while `llm-response-cache` is designed for production cost reduction with eviction and version tracking. For test-focused replay, use `llm-vcr`. For production caching with test-like benefits, use `llm-response-cache`.

### prompt-dedup

`prompt-dedup` normalizes prompts via text-level transformations (whitespace collapse, variable extraction, formatting normalization). Applying `prompt-dedup` normalization before hashing increases exact-match cache hit rates for prompts that differ only in formatting:

```typescript
import { normalize } from 'prompt-dedup';
import { createCache } from 'llm-response-cache';

const cache = createCache({
  normalizer: normalize,
});
// "What is the capital  of   France?" and "What is the capital of France?"
// now produce the same cache key.
```

### llm-dedup

`llm-dedup` coalesces identical in-flight LLM requests. `llm-response-cache` caches completed responses for future requests. The two are complementary: `llm-dedup` prevents duplicate concurrent requests (N simultaneous identical requests become 1 LLM call + N-1 in-flight dedup hits), while `llm-response-cache` prevents duplicate sequential requests (same request across time becomes 1 LLM call + all future cache hits). Layer `llm-dedup` on top of the cache-through wrapper:

```typescript
import { createDedup } from 'llm-dedup';
import { createCache } from 'llm-response-cache';

const cache = createCache({ maxEntries: 10_000 });
const cachedClient = cache.wrap(openai);
const dedupedClient = createDedup(cachedClient);

// All calls go through: dedup → cache → LLM
```

## 14. Testing Strategy

### Unit Tests

- **Cache key construction:** Test that identical requests produce the same key. Test that requests differing in any output-affecting parameter produce different keys. Test that requests differing only in non-output-affecting parameters (stream, user, api_key) produce the same key.
- **Canonicalization:** Test JSON key sorting at multiple nesting levels. Test whitespace normalization in message content. Test Unicode NFC normalization. Test handling of undefined/null parameters. Test numeric normalization (0 vs 0.0).
- **SHA-256 hashing:** Test with known input-output pairs. Test determinism across calls.
- **Model version detection:** Test extraction of version from response model field. Test extraction of date from model name patterns. Test version registry updates.
- **Invalidation strategies:** Test `auto` (entries deleted on version change). Test `soft` (entries flagged as stale, still served with warning). Test `manual` (no automatic action).
- **Cost computation:** Test per-hit cost calculation with known token counts and model prices. Test with exact usage from API response. Test with estimated tokens (character-based).
- **Token estimation:** Test the character-to-token approximation with known text-token pairs.
- **Stats tracking:** Test that hits, misses, tokensSaved, costSaved are incremented correctly. Test `resetStats`. Test model-filtered stats.
- **Eviction policies:** Test TTL expiry. Test LRU eviction at maxEntries. Test combined TTL + LRU. Test model-version-based eviction priority.

### Integration Tests

- **End-to-end cache hit:** Create cache -> set entry -> get with identical parameters -> verify hit with correct response.
- **End-to-end cache miss:** Create cache -> get with parameters not in cache -> verify null return.
- **Parameter sensitivity:** Test that changing temperature, model, max_tokens, tools, or response_format produces a different key and a cache miss.
- **Cache-through wrapper (OpenAI format):** Wrap a mock OpenAI client -> call chat.completions.create -> verify first call goes to client, second identical call returns from cache with `_cached: true`.
- **Cache-through wrapper (Anthropic format):** Same as above with Anthropic-style client.
- **Streaming cache miss:** Verify stream is forwarded in real time and response is cached on completion.
- **Streaming cache hit:** Verify cached response is replayed as a stream in all three modes (immediate, simulated, throttled).
- **Model version change (auto):** Populate cache with model version A -> trigger version change to B -> verify old entries are deleted.
- **Model version change (soft):** Populate cache -> trigger version change -> verify old entries are served with `stale: true`.
- **SQLite backend:** Create cache with SQLite -> set entries -> close -> reopen -> verify entries persist. Verify model-version invalidation works via SQL.
- **Redis backend:** Create cache with Redis -> set entries from process A -> read from process B -> verify shared cache.
- **Serialization round-trip:** Create cache -> add entries with various models and versions -> serialize -> deserialize -> verify all entries restored.

### Performance Tests

- **Hash computation speed:** Benchmark SHA-256 of canonicalized requests at various sizes (1 KB, 5 KB, 10 KB). Assert sub-100 microseconds for 10 KB.
- **Cache hit total latency:** Benchmark full pipeline (canonicalize + hash + lookup) for a cache hit. Assert sub-1ms for in-memory backend.
- **Cache throughput:** Benchmark hits per second for in-memory backend. Target: >100,000 hits/second.
- **Serialization speed:** Benchmark serialize/deserialize for 10,000 entries. Assert sub-1 second.
- **Memory usage:** Measure memory footprint for 1,000, 10,000, 50,000 entries. Assert linear scaling.

### Edge Case Tests

- **Empty cache:** `cache.get` returns null, `cache.stats` returns zeros.
- **Identical request different key order:** `{ model: "gpt-4o", temperature: 0 }` and `{ temperature: 0, model: "gpt-4o" }` produce the same key.
- **Default vs explicit parameters:** `{ temperature: undefined }` and `{}` produce the same key. `{ temperature: 1 }` and `{}` produce different keys.
- **Message content whitespace:** `"  Hello  world  "` normalized to `"Hello world"` for key computation (when normalizer is configured).
- **Unicode normalization:** Equivalent Unicode sequences (composed vs decomposed) produce the same key after NFC normalization.
- **Very long messages:** Messages with 100,000+ characters hash correctly and in reasonable time.
- **Concurrent access:** Multiple simultaneous `get` and `set` calls do not corrupt state.
- **Process shutdown:** `cache.close()` flushes pending writes for persistent backends.
- **Empty response:** A cached response with empty text is valid and returned correctly.
- **Tool call responses:** Responses containing tool calls (not just text) are cached and restored correctly.

## 15. Performance

### Cache Hit Latency Breakdown

| Step | Latency |
|---|---|
| Parameter canonicalization | < 0.05ms |
| SHA-256 hash computation | < 0.01ms |
| In-memory Map lookup | < 0.001ms |
| TTL/staleness check | < 0.001ms |
| Stats update | < 0.001ms |
| **Total cache hit (in-memory)** | **< 0.1ms** |

For comparison:
- LLM API call: 500-5,000ms
- Semantic cache hit (with local embedder): 2-10ms
- Semantic cache hit (with API embedder): 50-500ms

Exact-match cache hits are 1,000-50,000x faster than LLM API calls, and 20-100x faster than semantic cache hits. This is the primary advantage of exact matching over semantic matching.

### Cache Miss Latency Overhead

On a cache miss, the response cache adds latency for canonicalization and hashing (on lookup) and for storing the entry (after the LLM responds):

| Step | Added Latency |
|---|---|
| Canonicalization + hashing (lookup) | < 0.1ms |
| Entry storage (in-memory) | < 0.01ms |
| Entry storage (SQLite) | 0.5-2ms |
| Entry storage (Redis) | 1-5ms |
| **Total overhead on miss** | **< 0.2ms (memory), < 3ms (SQLite), < 6ms (Redis)** |

The miss overhead is negligible compared to the LLM call latency (< 0.04% of a 500ms call with in-memory storage).

### Memory Usage

| Entries | Avg Response Size | Entry Metadata | Total Memory |
|---|---|---|---|
| 1,000 | 2 KB | ~1 KB | ~3 MB |
| 10,000 | 2 KB | ~1 KB | ~30 MB |
| 50,000 | 2 KB | ~1 KB | ~150 MB |
| 100,000 | 2 KB | ~1 KB | ~300 MB |

Memory usage scales linearly with entry count and average response size. The dominant factor is response text storage. Entries with large responses (code generation, long-form content) consume proportionally more memory.

### Storage Backend Performance

| Backend | Write Latency | Read Latency | Notes |
|---|---|---|---|
| In-memory | < 0.01ms | < 0.01ms | Fastest, non-persistent |
| SQLite | 0.5-2ms | 0.1-0.5ms | Persistent, single-process |
| Redis | 1-5ms | 1-5ms | Persistent, multi-process |
| Filesystem | debounced | < 0.1ms (in-memory read) | Persistent, single-process |

## 18. Implementation Roadmap

### Phase 1: Core Cache (MVP)

1. **Cache key construction** (`key.ts`, `normalize.ts`). Canonicalization of request parameters, JSON serialization with sorted keys, SHA-256 hashing. Extensive unit tests with known input-output pairs verifying key determinism and parameter sensitivity.
2. **In-memory storage backend** (`storage/memory.ts`). `Map` for entries, doubly-linked list for LRU. O(1) get, set, delete, evict.
3. **Core cache logic** (`cache.ts`). `createCache`, `get`, `set`, `delete`, `clear`, `has`, `stats`. TTL checking, LRU eviction, hit/miss counting.
4. **Types** (`types.ts`). All interfaces and type definitions: `ResponseCacheOptions`, `CacheEntry`, `CacheHit`, `CacheStats`, `RequestParams`, `MessageInput`, `StorageBackend`.
5. **Public API exports** (`index.ts`). Export `createCache` and all types.
6. **Unit and integration tests** for all above.

### Phase 2: Model Version Tracking

1. **Version detection and tracking** (`version.ts`). Extract version from response model field, extract date from model name patterns, maintain version registry.
2. **Invalidation strategies** (`version.ts`). `auto` (delete stale), `soft` (flag stale), `manual` (no action). Integration with cache `get` and `set`.
3. **Tests** for version detection, registry updates, and all three invalidation strategies.

### Phase 3: Cache-Through Wrapper

1. **Proxy-based wrapper** (`wrapper/wrap.ts`). `cache.wrap(client)` returning a Proxy that intercepts calls.
2. **OpenAI adapter** (`wrapper/openai.ts`). Intercept `chat.completions.create`. Extract parameters, reconstruct response format on cache hit. Extract resolved model version from response.
3. **Anthropic adapter** (`wrapper/anthropic.ts`). Same for `messages.create`.
4. **Streaming support** (`wrapper/stream.ts`). Buffer-and-forward on miss. Replay on hit with configurable speed modes.
5. **Integration tests** with mock clients.

### Phase 4: Persistent Storage

1. **StorageBackend interface** (`storage/backend.ts`). Formalize the interface with `query` and `update` for version-based invalidation.
2. **SQLite backend** (`storage/sqlite.ts`). Schema, WAL mode, version-based invalidation queries.
3. **Filesystem backend** (`storage/filesystem.ts`). JSON file, debounced writes, in-memory lookup.
4. **Integration tests** for persistence, restart recovery, invalidation.

### Phase 5: Advanced Features

1. **Redis backend** (`storage/redis.ts`). Redis hashes for entries, TTL via `PEXPIRE`, shared cache across processes.
2. **Cost tracking** (`stats.ts`). Token estimation, model pricing, `model-price-registry` integration, per-model stats.
3. **Serialization** (`serialization.ts`). Binary format for cache export/import.
4. **CLI** (`cli.ts`). `stats`, `clear`, `invalidate`, `export`, `import` commands.

### Phase 6: Polish and Documentation

1. **README.md** with quick-start guide, API reference, and integration examples.
2. **Performance benchmarks** published in README.
3. **Edge case hardening** based on real-world usage patterns.

## 19. Example Use Cases

### Deterministic Batch Processing

An evaluation pipeline tests 200 prompts against GPT-4o with `temperature: 0`. Each evaluation run costs $2.00. The pipeline runs 20 times per day during development iteration.

```typescript
import { createCache } from 'llm-response-cache';
import OpenAI from 'openai';

const openai = new OpenAI();
const cache = createCache({
  storage: { type: 'sqlite', path: './eval-cache.db' },
  maxEntries: 100_000,
  invalidationStrategy: 'auto',
});

const cachedClient = cache.wrap(openai);

for (const prompt of evaluationPrompts) {
  const response = await cachedClient.chat.completions.create({
    model: 'gpt-4o',
    temperature: 0,
    messages: [{ role: 'user', content: prompt }],
  });
  evaluateResponse(prompt, response);
}

// First run: 200 API calls, $2.00, ~5 minutes
// Subsequent runs: 200 cache hits, $0.00, ~0.1 seconds
// Daily savings: 19 runs * $2.00 = $38.00/day
// Annual savings: $13,870
```

### Layered Caching (Exact + Semantic)

For maximum cache hit rates, layer an exact-match cache in front of a semantic cache. The exact cache handles verbatim repeats (sub-millisecond, zero overhead). The semantic cache handles paraphrases (milliseconds, requires embedding).

```typescript
import { createCache as createExactCache } from 'llm-response-cache';
import { createCache as createSemanticCache } from 'llm-semantic-cache';

const exactCache = createExactCache({
  maxEntries: 50_000,
  storage: { type: 'sqlite', path: './exact-cache.db' },
});

const semanticCache = createSemanticCache({
  embedder: localEmbedder,
  threshold: 0.92,
  maxEntries: 10_000,
  storage: { type: 'sqlite', path: './semantic-cache.db' },
});

async function smartCache(messages, model, params) {
  // Layer 1: Exact match (< 0.1ms)
  const exact = exactCache.get(messages, model, params);
  if (exact) return exact.response;

  // Layer 2: Semantic match (2-10ms with local embedder)
  const prompt = serializeMessages(messages);
  const semantic = await semanticCache.get(prompt, { model });
  if (semantic) return semantic.response;

  // Layer 3: LLM call (500-5000ms)
  const response = await callLLM(messages, model, params);
  exactCache.set(messages, model, params, { text: response });
  await semanticCache.set(prompt, response, { model });
  return response;
}

// Combined hit rate: exact catches ~40% (verbatim repeats)
// Semantic catches additional ~25% (paraphrases)
// Total hit rate: ~65% vs ~40% (exact only) or ~55% (semantic only)
```

### Model Migration with Version Tracking

When migrating between model versions, the cache automatically detects the version change and handles invalidation according to the configured strategy.

```typescript
const cache = createCache({
  storage: { type: 'sqlite', path: './production-cache.db' },
  invalidationStrategy: 'soft', // Serve stale but flag it
});

const cachedClient = cache.wrap(openai);

// Before model update: gpt-4o resolves to gpt-4o-2024-08-06
// Cache is populated with responses from gpt-4o-2024-08-06

// After OpenAI updates gpt-4o to gpt-4o-2024-11-20:
// First cache miss detects version change via response metadata
// Existing entries flagged as stale (soft invalidation)
// Stale entries served with stale: true flag
// Application can log staleness or refresh in background

// Later, when ready to fully migrate:
cache.invalidate('gpt-4o'); // Remove all stale gpt-4o entries
// Cache repopulates with responses from new model version
```
